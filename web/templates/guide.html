01 — Getting Started

First Login

When you first access SIEM-Lite, you'll log in with default credentials. For security, you are required to change your password immediately — the system will redirect you to the Account page until you do.

Default credentials: admin / admin123
These only work once. After your first login, set a strong password.

Navigation

The left sidebar organizes the interface into three sections:

02 — Dashboard

The dashboard is your security overview. It shows:

  • KPI Strip — Total events, open alerts, active rules, threat indicators. The "Open Alerts" card uses inverted (black) styling for emphasis.
  • Events by Severity — Horizontal bar chart showing the distribution of event severity levels.
  • Events by Type — Breakdown by category (authentication, network, firewall, etc.)
  • Pipeline Throughput — How many events have been ingested, processed, stored, and any errors.
  • Recent Alerts — The latest alerts generated by detection and correlation rules.
The dashboard auto-refreshes every 30 seconds. Click Refresh for an immediate update.

03 — Events

The Events page shows every log entry that has been ingested and normalized. Each row represents one event with:

  • Timestamp — When the event occurred
  • Severity — Critical, High, Medium, Low, or Info
  • Type — Event classification (authentication, network, etc.)
  • Source / Destination — IP addresses involved
  • Action — What happened (logon, deny, connect, etc.)
  • Message — Human-readable description
  • Risk Score — 0–100 computed risk score with a visual bar

Filtering

Use the filter bar at the top to narrow events by text, type, or severity. Click any row to see full event details including raw log data, enrichment info, threat intel matches, and MITRE ATT&CK techniques.

04 — Alerts

Alerts are generated when detection or correlation rules fire. Each alert contains:

  • Title & Description — What was detected
  • Severity — How urgent it is
  • Source Events — The events that triggered it
  • Risk Score — Aggregated risk level
  • Evidence — Supporting data

Alert Workflow

Alerts follow a lifecycle you control:

OPEN → ACKNOWLEDGED → RESOLVED → CLOSED
                ↘ ESCALATED ↗
ActionWhat it means
AcknowledgeYou've seen it and are looking into it
ResolveYou've investigated and handled it
EscalateSend to a higher-tier analyst

05 — Search

The Search page lets you query the entire event store with a field-level syntax.

Query Syntax

SyntaxExampleMatches
field:valuesource_ip:192.168.1.1Exact match
field:a,b,cseverity:high,criticalAny of these values
-field:value-action:logoffNOT this value
field:/regex/message:/failed.*password/Regex match
"text""failed password"Full-text search

Common Queries

# All failed authentications
action:logon_failed

# Events from a specific IP
source_ip:45.227.255.206

# Critical or high severity events
severity:critical,high

# Web attacks
message:/sql|injection|xss|traversal/

# Events with threat intel matches
ioc_matched:*

# PowerShell execution
message:powershell event_type:process

06 — Detection Rules

SIEM-Lite ships with 37 built-in rules covering common attack patterns. Each rule has a type:

TypeHow it worksExample
DetectionSingle event matchFailed SSH login
CorrelationMultiple events in a time window10 failed logins = brute force
Threat IntelMatch against known-bad indicatorsConnection to malicious IP
AnomalyStatistical deviation from baselineUnusual login time

Creating a Rule

  1. Go to Rules → New Rule
  2. Set name, severity, type, and category
  3. Define conditions as JSON: [[{"field": "action", "operator": "eq", "value": "logon_failed"}]]
  4. For correlation rules, set the window (seconds) and threshold (count)
  5. Save — the rule is active immediately

Operators

eq, ne, gt, gte, lt, lte, contains, startswith, endswith, in, not_in, regex, exists, cidr

07 — Threat Intelligence

The Threat Intel page manages Indicators of Compromise (IOCs). Every incoming event is automatically checked against these indicators.

Indicator Types

  • IP — Malicious IP addresses
  • CIDR — Entire malicious network ranges
  • Domain — Malicious domain names
  • URL — Malicious URLs
  • Hash — Known-malicious file hashes
  • Email — Malicious email addresses

Adding Indicators

Click + Add Indicator, choose the type, enter the value, set severity and tags. The indicator is immediately active — the next event matching it will generate an alert.

The system comes pre-loaded with sample indicators including known malicious IPs, Tor exit nodes, and suspicious domains.

08 — Ingesting Real Logs

Method 1: API (simplest)

curl -X POST http://localhost:8443/api/events \
  -H "Content-Type: application/json" \
  -d '{"raw_data": "<34>Oct 11 22:14:15 host sshd: Failed password"}'

Method 2: Webhook

curl -X POST http://localhost:8443/api/ingest/webhook \
  -d 'src=1.2.3.4 dst=10.0.0.1 action=deny'

Method 3: Log Simulator

Generate realistic events continuously for testing:

python scripts/ingest_simulator.py --rate 10

Method 4: Real Collectors

Edit config/siem.yaml to enable collectors, then start with --collectors:

collectors:
  syslog_udp:
    type: syslog
    protocol: udp
    host: 0.0.0.0
    port: 514
    enabled: true

  file_logs:
    type: file
    directory: /var/log
    pattern: "*.log"
    enabled: true

Supported Log Formats

The system auto-detects and parses: Syslog (RFC 3164/5424), JSON, CEF, LEEF, Apache/Nginx, Windows Events, CSV/TSV, and custom regex/Grok.

10 — Security Best Practices

Account Security

  • Change default passwords immediately — Go to Account → Change Password
  • Use strong passwords — Minimum 8 chars with upper, lower, numbers, and special characters
  • Create separate users — Don't share the admin account; create analyst/viewer accounts
  • Generate API keys — Use API keys instead of passwords for automated integrations

System Security

  • Set a secret key — Configure security.secret_key in config/siem.yaml
  • Enable HTTPS — Set security.require_https: true and use a reverse proxy with TLS
  • Restrict network access — Bind to 127.0.0.1 if only local access is needed
  • Review the audit log — Check Admin → Audit Log regularly
For production deployments: Always run behind a reverse proxy (nginx/Apache) with TLS enabled. Never expose the Flask dev server directly to the internet.

11 — Production Deployment

Systemd Service (Linux)

[Unit]
Description=SIEM-Lite Log Correlation Engine
After=network.target

[Service]
Type=simple
User=siem
WorkingDirectory=/opt/siem-lite
ExecStart=/usr/bin/python3 run.py --port 8443
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target

Nginx Reverse Proxy (TLS)

server {
    listen 443 ssl;
    server_name siem.yourorg.com;

    ssl_certificate     /etc/ssl/certs/siem.crt;
    ssl_certificate_key /etc/ssl/private/siem.key;

    location / {
        proxy_pass http://127.0.0.1:8443;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

Docker

docker build -t siem-lite .
docker run -d -p 8443:8443 -v siem-data:/app/data siem-lite
